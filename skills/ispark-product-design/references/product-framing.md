# Product Framing

Ask:

- who has the problem
- what current workaround exists
- what outcome proves value
- how often the workflow happens
- how urgent the problem is when it appears
- what distribution path makes adoption plausible
- what is the smallest useful workflow
- what should not be built yet

For founder-style critique, challenge demand, distribution, workflow frequency, urgency, and whether the solution is narrower than the problem.

Separate durable product capability from implementation detail. Do not treat legacy database shape, internal pipeline structure, or one product workspace as the product requirement unless the user explicitly asks for compatibility.

Before implementation, state the product assumption, the non-goal, and the simplest coherent workflow. If multiple interpretations would change product scope, ask or record the decision point instead of silently choosing.
