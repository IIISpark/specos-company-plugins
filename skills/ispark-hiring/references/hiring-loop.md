# 招聘闭环方法：从组织定位到面后决策

用途：帮助岗位发起人、HR、面试官及其 Agent 把招聘从零散的“看简历、凭感觉面试”，转成一条可追溯、可复用、能持续校准的证据链。

这是一份 meta-document。它规定如何生成招聘过程中的各类文档和判断，不包含任何具体候选人的材料，也不替代 human owner 的录用决策、HR 制度或合规要求。

## 1. 最终要建立的不是题库，而是决策闭环

招聘的主链是：

```text
组织与岗位发起人定位
  -> 当前责任缺口
  -> 岗位任务与成功结果
  -> 评价维度与证据标准
  -> 批次初筛与待核验假设
  -> 候选人专属面试 runbook
  -> 面试事实与证据归档
  -> 面后评估与 human 决策
  -> 岗位画像、题目和流程校准
```

上游没有定义清楚，下游就不应猜：

- 不知道团队为什么招人，就不能先写一个泛化 JD。
- 不知道这个人入职后要交付什么，就不能只按技术关键词排简历。
- 没有预先定义证据维度，就不能面后临时改变评价标准。
- 没有候选人的真实证据，就必须标记 `Unknown`，不能用学校、公司名、表达风格或相似经历代替。
- 没有 human owner 拍板，Agent 不能把建议写成最终录用或拒绝决定。

## 2. 核心原则

### 2.1 从工作出发，不从理想人格出发

先定义要转移的责任、要解决的瓶颈和可验证结果，再定义人。不要以“聪明”“同频”“靠谱”“有潜力”这类无法直接校准的词作为画像终点。

抽象词必须被翻译成工作证据。例如：

| 抽象词 | 可核验的工作证据 |
| --- | --- |
| 可靠 | 会按约定同步进展；遇到阻塞会暴露事实、尝试和需要的支持；交付物可复查 |
| 学习快 | 能说明陌生问题的学习路径，并把学习转成代码、文档、实验或可验证结论 |
| 工程能力强 | 能讲清本人负责的系统边界、失败、排障、测试、发布和维护闭环 |
| 协作好 | 能区分讨论权、建议权和最终决策权；不同意见能用证据推进并执行已拍板结论 |
| 有 ownership | 能明确个人贡献、未完成项和风险，不把团队或 Agent 的产出全部归为自己 |

### 2.2 先定义维度，再看候选人

权重、硬条件和评分锚点必须在进入候选人比较前冻结。候选人可以暴露画像问题，但不能为了喜欢或不喜欢某个人而回改当轮标准。

### 2.3 简历只生成假设，不生成能力事实

简历、作品介绍和公开主页决定“值得核验什么”，不直接证明候选人具备某项能力。最终证据来自可追溯的项目细节、现场演示、代码或作品走查、失败与修复过程、工作样本以及经授权的事实核验。

### 2.4 `Unknown` 不等于低分

没有证据时记录 `Unknown`，并决定下一阶段是否值得花成本核验。只有出现与维度直接相关的反证，才给低评价。这样可以避免把“简历没写”“面试没问到”和“候选人不会”混成一个结论。

### 2.5 区分人、岗位和阶段

面后结论至少分开回答：

- 候选人已证明的能力是什么。
- 候选人对当前岗位和当前团队阶段是否匹配。
- 哪些潜力值得未来或其他岗位重新评估。

“当前不推进”不等于对人的总体价值判断。

### 2.6 Agent 负责压缩上下文，human 负责承担决定

Agent 可以抽取、归一化、对照、生成问题、整理证据和发现矛盾；岗位 owner 与 HR 对岗位画像、硬条件、面试边界、候选人沟通和最终决定负责。

## 3. 证据与隐私边界

### 3.1 证据层级

| 等级 | 定义 | 招聘用途 |
| --- | --- | --- |
| A | 现场可复现的代码、作品、任务结果、逐字稿中的具体事实，或经授权核验的原始记录 | 可以直接支撑判断 |
| B | 可归属、可定位的公开作品和持续维护记录 | 可以支撑判断，但仍需确认个人贡献边界 |
| C | 简历、自述、第三方转述或未经复核的指标 | 只能形成待核验假设 |
| D | 从学校、公司、口音、表达风格、照片、社交关系等做出的推测 | 不得作为判断证据 |

每条关键判断建议保留最小证据单元：

```yaml
claim: 候选人被认为具备或缺少什么
dimension: 对应哪个预定义维度
source: 来源及可定位位置
observation: 实际观察到什么，不写人格诊断
strength: strong | moderate | weak | counterevidence | unknown
ownership: 候选人本人、团队、Agent 或第三方分别贡献了什么
open_questions: 还缺什么证据
```

### 3.2 隐私和公平性底线

- 只处理岗位需要且已获授权的候选人资料，使用最小必要字段。
- 原始简历、联系方式、逐字稿、背调和个人评价留在受控的人事或候选人空间；公司方法文档只放结构和模板。
- 不把候选人个人信息提交给未授权的外部搜索、模型、网页或第三方服务。
- 不推断或使用与岗位无关的敏感特征，也不使用学校、住址、照片等作为其代理信号。
- 所有硬条件必须与岗位真实约束相关，由岗位 owner 与 HR 预先确认，并对同一批候选人一致执行。
- 记录可观察行为、事实和工作影响，不写羞辱性标签或未经证实的人格结论。
- 需要候选人联系、背调、薪酬、offer、拒绝或对外发布时，Agent 必须停在草稿状态，交由授权的人执行。

## 4. 状态机与交付物

| 状态 | 必要输入 | 必须产物 | 进入下一状态的 gate |
| --- | --- | --- | --- |
| `context-ready` | 公司/团队目标、岗位发起人责任、当前瓶颈 | Hiring Context Brief | 说清为什么现在招、什么责任要转移、哪些责任不转移 |
| `role-ready` | Context Brief、实际任务、管理带宽 | Role Brief | 前 90 天结果、must-have、可培养项、non-goals 和硬约束已明确 |
| `rubric-ready` | Role Brief | Evidence Rubric | 维度、权重、评分锚点、硬条件、证据来源和无效代理已冻结 |
| `screened` | 同批候选人材料、Rubric | Batch Screening + Candidate Cards | 每个排序都有岗位证据、风险和下一步核验问题 |
| `prechecked` | Candidate Card、HR 已确认约束 | HR Precheck Record | 候选人已由授权 HR 确认真实岗位条件，没有用无关敏感信息替代 |
| `interview-ready` | Candidate Card、通用题、候选人材料 | Interview Runbook | 每个问题都映射到维度或证据缺口，并有时间与停止条件 |
| `interviewed` | 面试、作品走查或工作样本 | 原始记录 + Interview Notes | 原始事实与面试官即时印象分开保存 |
| `evaluated` | 原始记录、Rubric、Runbook | Post-interview Evaluation | 每项评分可回到证据；`Unknown` 与反证分开 |
| `decided` | Evaluation、岗位约束、横向比较 | Human Decision Record | 授权 owner 确认动作、理由、风险和后续核验 |
| `closed/handed-off` | Decision Record | HR 动作或 onboarding 验证项 | 资料按权限归档；方法改进回写下一轮版本 |

任何 gate 不满足时，Agent 输出缺失字段和影响，停在当前状态。不要通过默认值把流程伪装成已经完成。

### 4.1 建议的受控工作区结构

下面是逻辑结构，不要求所有团队使用同一个系统。它应存在于 HR 或岗位 owner 授权的招聘空间，不能放进本 skill、公司方法源码或其他公开文档节点：

```text
<hiring-space>/<role-slug>/<round-id>/
  context-brief.md
  role-brief.md
  evidence-rubric.md
  batches/
    <batch-id>/screening.md
  candidates/
    <candidate-id>/
      source-index.md
      screening-card.md
      hr-precheck.md
      interview-runbook.md
      interview-notes.md
      post-interview-evaluation.md
      decision-record.md
  process-retrospective.md
```

规则：

- `<candidate-id>` 使用稳定内部 ID；展示姓名和联系方式只保留在访问受控的 source index 或 HR 系统。
- 原始简历、逐字稿和作品不复制进分析文档，只记录授权来源及可定位引用。
- `role-brief` 和 `evidence-rubric` 标记版本；Candidate Card、Runbook、Evaluation 与 Decision Record 都记录所用版本。
- 面试官即时笔记与事后评估分开，避免后续整理覆盖原始观察。
- 归档、保留期限和删除规则服从公司的候选人数据政策；方法文档不自行发明期限。

## 5. Step 0：校准组织与岗位发起人

招聘首先是在回答“现有系统为什么需要新增一个人”。岗位发起人先完成以下问题：

1. 团队当前目标和阶段是什么？
2. 发起人现在亲自承担哪些责任？哪些是最终决策，哪些是执行负担？
3. 哪些工作已经成为瓶颈，有什么可观察证据？
4. 新人加入后，团队希望哪部分变轻或变快？
5. 哪些责任在当前阶段不能移交？
6. 团队能提供多少 onboarding、review 和培养带宽？
7. 如果三个月仍无人加入，真实损失是什么？如果招错人，退出成本是什么？

输出模板：

```markdown
# Hiring Context Brief

## 当前阶段与目标

## 岗位发起人的责任边界
- 最终负责：
- 可以共同决策：
- 希望转移的执行责任：

## 当前瓶颈及证据

## 团队可提供的培养与管理带宽

## 为什么现在招聘

## 不招聘和招错人的成本

## 本轮 non-goals
```

## 6. Step 1：把责任缺口翻译成岗位

岗位画像不从 title 开始，从“入职后要完成的工作”开始。

### 6.1 Role Brief 必须回答

- 岗位使命：这个角色存在是为了改变什么事实。
- 关键责任：稳定承担哪些工作，不承担哪些工作。
- 前 30/60/90 天结果：以 artifact、系统状态、客户结果或团队责任变化表达。
- 必须已有：没有就无法在当前团队成功的能力或条件。
- 可以培养：入职后能在现有带宽内补齐的能力。
- 加分项：只提高优先级，不伪装成硬条件。
- 失败画像：不是写人格标签，而是写会导致岗位失败的可观察工作模式。
- 真实约束：地点、投入时间、用工形态、协作时区等由 HR 确认的岗位条件。
- Non-goals：避免把未来可能需要的能力全部塞进当前岗位。

```markdown
Version: <role_version>

<入职后什么事实应成立>

<当前责任缺口和证据>

-

- 岗位 owner：
- HR owner：
- 面试参与者：
```

## 7. Step 2：建立评价维度和证据 Rubric

### 7.1 维度来自岗位，不照抄历史权重

常见维度可以作为问题清单，但每个岗位要重新确认：

- 诚信与贡献边界：陈述是否可追溯，是否区分本人、团队、Agent 和第三方贡献。
- 工作与交付能力：能否把任务推进到可运行、可验证、可维护的结果。
- 学习与迁移能力：能否从陌生问题形成模型、实验和复盘。
- 沟通与协作可靠性：能否暴露风险、处理分歧、接受 review 并维护共同上下文。
- 相关领域能力：对当前岗位核心问题的理解和实践深度。
- 投入条件与岗位约束：可用时间、地点、用工方式等是否满足已确认的真实约束。
- 成长空间：在团队可提供的培养带宽内，是否可能承担更完整的责任。

不要使用笼统的“文化匹配度”作为兜底项。需要评价的内容应拆成可观察的协作行为和岗位约束。

### 7.2 Rubric 模板

```markdown
Role version: <role_version>
Rubric version: <rubric_version>

| 维度 | 权重 | 必须回答的问题 | 强证据 | 反证 | 无效代理 | 核验方式 |
| --- | ---: | --- | --- | --- | --- | --- |
| <维度> | <0-100> |  |  |  |  |  |
```

建议评分锚点：

- `5`：存在多个相互支持、可复查的强证据，候选人能解释边界与失败。
- `4`：有直接相关证据，深度和 ownership 基本清晰。
- `3`：有部分相关证据，但规模、独立性或迁移性仍需验证。
- `2`：只有弱相关或间接证据，且出现明显能力缺口。
- `1`：存在直接反证，无法满足该维度的关键要求。
- `Unknown`：本轮没有获得足够证据，不参与机械平均，单独决定是否继续核验。

硬条件与加权评分分开。硬条件必须在看候选人前定义；评分不能抵消已经确认的硬条件失败。

## 8. Step 3：批次初筛

初筛目标不是录用，而是分配下一步核验成本。

### 8.1 标准动作

1. 固定批次、来源和截止时间，避免候选池不断变化却沿用旧排序。
2. 对每位候选人抽取与 Role Brief 直接相关的经历、成果声明和约束信息。
3. 区分候选人声明、公开可核验事实和 Agent 推断。
4. 记录最强信号、核心风险、证据缺口和下一轮最值得问的问题。
5. 按“值得投入多少核验成本”分流，不把 `P0/P1/P2` 写成能力等级。
6. 同批候选人使用同一版 Rubric；若岗位画像变更，明确开新版本并重新评估全批次。

### 8.2 Candidate Screening Card

先用批次表固定比较范围和队列，再进入单人卡片：

```markdown

Sources and cutoff: <来源与截止时间>

| Candidate ID | Routing | Strongest role evidence | Biggest gap | Next action |
| --- | --- | --- | --- | --- |
```

### 8.3 HR 前置确认

HR 前置确认用于节省候选人和面试官时间，但不能变成收集无关个人信息的入口。

- 只确认 Role Brief 中已经声明的真实约束，例如用工形态、可投入时间、工作地点、到岗窗口和已批准的薪酬区间是否存在根本冲突。
- 候选人沟通由授权 HR 或岗位 owner 执行；Agent 只起草最小问题清单或整理 HR 已确认结果。
- 进入技术或业务面试的记录只需要 `met / unmet / unknown` 和必要说明，不扩散候选人的敏感原话。
- 若约束本身有争议，先回到 Role Brief 与 HR policy，不让单个候选人承担标准不清的后果。

```markdown
# HR Precheck Record - <candidate_id>

Role version: <role_version>

| 已声明岗位约束 | 状态 | 最小必要说明 | Human owner |
| --- | --- | --- | --- |
| <约束> | met / unmet / unknown |  |  |
```

## 9. Step 4：把证据缺口变成面试 runbook

一份好的面试稿是“待核验假设的执行计划”，不是把通用题库复制给所有人。

### 9.1 题目生成公式

```text
候选人声明
  -> 对应岗位维度
  -> 当前缺失或冲突的证据
  -> 主问题
  -> 逐层追问
  -> 通过/不足/停止信号
```

通用追问阶梯：

1. `Context`：问题、用户、约束和规模是什么？
2. `Ownership`：你本人、团队和 Agent 分别做了什么？
3. `Mechanism`：数据流、状态、接口或实现机制是什么？
4. `Failure`：真实失败过什么？如何定位？
5. `Evidence`：哪条代码、测试、指标、记录或结果能证明？
6. `Trade-off`：为什么选这个方案，放弃了什么？
7. `Reflection`：现在重做会改什么？

当候选人已经无法说明 ownership 或基本机制时，记录证据不足并切题。不要在现场持续给出标准答案，再把跟随回答当成候选人能力。

### 9.2 Runbook 结构

```markdown
# Interview Runbook - <candidate_id> - <role>

Role version: <role_version>
Rubric version: <rubric_version>

## Interview outcome
<本场结束时必须减少哪些不确定性>

## Pre-interview hypotheses
| 假设 | 来源 | 对应维度 | 需要的证据 |
| --- | --- | --- | --- |

## Common baseline block
```

## 10. Step 5：面后评估

### 10.1 先还原事实，再评分

1. 以录音、逐字稿、面试官笔记、现场代码或工作样本为事实源。
2. 修正自动摘要中的姓名、时间、数字或归属错误，并记录冲突。
3. 抽取强证据、弱证据、反证和 Unknown，映射回当轮 Rubric。
4. 评分必须附证据；不能只写“感觉不错”或“气场不合”。
5. 分开写能力、当前岗位匹配、培养成本和长期可验证项。
6. 给出下一步动作，但最终决定由授权的 human owner 确认。

### 10.3 Decision Record

建议使用有限、可行动的标签：

- `advance`：现有证据足以进入下一阶段。
- `advance-after-validation`：先补一个明确证据再推进。
- `hold`：当前优先级或岗位时机不合适，保留重新评估条件。
- `redirect`：当前岗位不匹配，但有证据支持另一个明确方向。
- `no-advance`：当前不继续，写清岗位相关原因。

Decision Record 至少包含：决策人、日期、当轮 Role/Rubric 版本、动作、三条核心证据、最大风险、仍未知项和对 HR 的最小必要口径。

## 11. Step 6：让个案反馈方法，而不是污染方法

每轮结束后分别复盘三层：

- **岗位层**：所有候选人都在同一处错位，是否说明岗位 title、来源渠道或 must-have 定义有误？
- **流程层**：哪些问题高效获得证据，哪些问题在教学、重复或引导？
- **候选人层**：当前动作是什么，后续重开需要什么新证据？

方法更新规则：

- 单个候选人的特殊性先记为个案，不立即变成普遍规则。
- 重复出现且影响判断质量的问题，才进入下一版 Role Brief、Rubric 或通用题。
- 新版本从下一批开始使用；如果必须用于当前批次，应重新检查所有候选人，不能只重评某个人。
- 面试结论不能反向伪造初筛标准。保留“面前判断”和“面后证据”的差异，才能改进筛选质量。

## 12. Agent 执行协议

### 12.1 Agent 可以做

- 从授权材料中抽取岗位相关事实并标注来源。
- 把组织责任缺口整理成 Role Brief 草稿。
- 生成 Evidence Rubric 草稿并检查抽象词是否可观测。
- 使用同一 Rubric 整理候选人批次和证据缺口。
- 为单个候选人生成假设驱动的面试 runbook。
- 从逐字稿生成证据映射、面后评估草稿和流程复盘。
- 发现来源冲突、评分漂移、提示污染和隐私风险。

### 12.2 Agent 不可以做

- 静默补全岗位目标、硬条件、候选人经历或个人贡献。
- 根据姓名、照片、学校、住址、家庭、健康或其他无关特征推断能力和稳定性。
- 把简历关键词、学历、前雇主或表达流畅度当成能力结论。
- 未经授权联系候选人、索取材料、执行背调、发 offer、发拒信或公开评价。
- 把个人系统中的原始候选人判断直接复制进公司文档。
- 在不同候选人之间使用未声明的不同标准。

### 12.3 Agent Stop Rules

出现以下情况时停止评价并请求 human owner：

- Hiring Context Brief、Role Brief 或 Rubric 缺失，且继续会导致静默选边。
- 岗位 owner 与 HR 对硬条件、职责或评价标准存在冲突。
- 资料权限、候选人同意范围或数据存放边界不清楚。
- 需要推断敏感特征、人格、健康、家庭或其他与岗位无关的信息。
- 原始材料互相冲突，且冲突会改变候选人结论。
- 需要对外联系、发布、背调、薪酬、offer、拒绝或其他真实人事副作用。

## 13. 给同事 Agent 的启动 Brief

下面这段可以作为一轮招聘工作的起始 prompt。先填空，再把对应事实源路径交给 Agent；不要把密码、token 或无关候选人材料放入 prompt。

```markdown
你正在协助 <团队/岗位 owner> 完成 <岗位> 的招聘闭环。

Outcome:
- 从组织责任缺口出发，形成可追溯的 Role Brief、Evidence Rubric、候选人筛查、面试 runbook 和面后评估。
- 所有结论能回到授权来源；Unknown 不得被猜测填充。

Context:
- 团队目标：<...>
- 岗位 owner 责任：<...>
- 希望转移的责任：<...>
- 当前瓶颈证据：<...>
- 可用培养/管理带宽：<...>

Inputs:
- 公司/团队文档：<路径>
- 当前 Role Brief：<路径或“待生成”>
- 当前 Rubric：<路径或“待生成”>
- 候选人授权材料：<路径>
- HR 已确认约束：<...>

Constraints:
- 只使用岗位相关、已授权材料。
- 区分事实、候选人声明、Agent 推断和 Unknown。
- 不使用或推断与岗位无关的敏感特征。
- 不联系候选人，不执行背调、offer、拒绝或公开发布。
- 任何评分必须附来源；最终决定必须留给 human owner。

Non-goals:
- 不重写组织架构，不扩大岗位范围，不评价候选人的总体人格或人生价值。

Deliverables:
1. 当前状态与缺失输入。
2. 本阶段要求的标准产物。
3. 证据冲突、风险和 Unknown。
4. 下一 gate、owner 和停止条件。

Acceptance criteria:
- 产物符合招聘闭环方法中的模板。
- 同批候选人使用同一版 Role Brief 与 Rubric。
- 候选人结论可从评估回溯到原始来源和面试问题。

Stop rules:
- 遇到方法文档所列 Agent Stop Rules 时立即停止并报告。
```

## 14. 完成检查表

一次完整招聘闭环至少满足：

- [ ] 能从岗位结论一路回溯到组织责任缺口。
- [ ] 前 30/60/90 天结果可验证，must-have 与可培养项已分开。
- [ ] Rubric 在候选人比较前冻结，硬条件与评分分开。
- [ ] 所有候选人使用同一版基线，定制问题只用于核验各自证据缺口。
- [ ] 简历声明、公开事实、现场证据和 Agent 推断已明确分层。
- [ ] `Unknown` 没有被当作低分或默认能力。
- [ ] 面试稿的每个核心问题都对应维度、假设或风险。
- [ ] 面后评分附证据，并区分能力、当前岗位匹配和培养成本。
- [ ] 最终决定、候选人联系和人事副作用由授权 human owner 执行。
- [ ] 公司方法文档不含候选人 PII、私有逐字稿或个人系统中的原始评价。
- [ ] 本轮流程改进只影响下一版，未对单个候选人追溯性改标。

当这些条件成立时，另一个同事的 Agent 不需要知道方法最初来自谁，也能沿着同一条证据链完成可审查的招聘辅助工作。
